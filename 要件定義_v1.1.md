## 1. プロジェクト概要

### 1.1 誰デザとは

「誰デザ」は「誰がデザインしたかを当てるゲーム」の略で、イラスト界隈の身内向けお題企画である。基本的な流れは以下の6ステップで構成される。

1. **募集**：主催がテーマ（お題）を決めて参加者を募集する。
2. **デザイン**：各参加者がテーマに沿ってキャラクターをデザインする。
3. **シャッフル交換**：提出されたデザインをシャッフルして参加者間で交換する。このとき自分のデザインが自分に割り当てられないようにする。
4. **作画**：受け取ったデザインを、そのデザインを描いた本人とは別の参加者が作画（清書・イラスト化）する。
5. **一覧公開**：完成した作画を一覧で公開する。
6. **投票・結果発表**：「どの作品を誰がデザインしたか」を参加者・観覧者が投票で当て、最後に答え合わせと結果を発表する。

規模は1企画あたり数人〜十数人程度の小規模なコミュニティ企画である。

### 1.2 背景と課題

現状、主催者は専用ツールを持たず、以下のような複数の汎用ツールを継ぎはぎして運営している。

- **X（旧Twitter）**：告知・募集・投票・結果発表
- **Excel の乱数機能**：シャッフル（誰に誰のデザインを割り当てるか）の抽選
- **TwiPla / Privatter**：参加者募集・限定公開
- **Discord**：参加者への連絡・提出物の受け渡し
- **X アンケート**：作者当て投票

この運用には次の問題がある。

- **主催の運営負荷が高い**：ツールをまたいで手作業でデータを転記・管理する必要があり、参加者が増えるほど負担が増す。特にシャッフル割当は手作業だと「自分に当たらない」条件を担保するのが煩雑でミスも起きやすい。
- **匿名性の担保が難しい**：作者が事前にバレると企画そのものが成立しない。しかし現状のツールでは提出物と作者の紐付けが主催の手元に露出しやすく、うっかり漏れや、ツール仕様上の匿名性の弱さがリスクになる。
- **情報が分散する**：企画の進行状況・提出物・投票結果がツールごとに散らばり、参加者も主催も全体像を追いにくい。

### 1.3 目的

本サイトは、誰デザ企画の運営に必要な機能を1箇所に統合し、次の2大テーマを実現する。

1. **匿名性の担保**：作者と提出物の紐付けをシステム側で厳格に管理し、投票・結果発表まで作者が露出しないようにする。
2. **主催の運営負荷軽減**：募集・提出・シャッフル・作画・投票・結果発表という一連の流れを、ツールを横断せずサイト内で完結できるようにする。

### 1.4 設計原則：デザイナーと作画者の分離が匿名性を生む（重要）

本企画で当てる対象は **デザイナー（③のデザイン提出者）** であって、作画者ではない。作画の絵柄は「作画者」を示唆するため、デザイナーの推理からむしろ**ミスリードする防御として機能**する。したがって、

- **デザイナーの匿名性**と**作画者の匿名性**の両方を、結果発表フェーズまで守る必要がある。
- 割当（デザイン⇔作画者）の対応が漏れると、絵柄からデザイナーが逆算されうるため、割当情報の秘匿は匿名性の生命線である。

この「デザイナーと作画者を分離し、両者を結果発表まで秘匿する」構造が、企画を成立させる核心的な仕組みである。以降の要件（特に4.3・第6章・第7章）はすべてこの原則に従う。

## 2. ターゲットユーザー

本サイトの利用者は「主催者」「参加者」「観覧者」の3種類である。

### 2.1 主催者（オーガナイザー）

**ペルソナ**：イラストコミュニティで企画を回すのが好きな個人。X を中心に活動し、数人〜十数人規模の身内企画を不定期に主催している。デザインスキルはあるが、プログラミングやデータ管理は専門外。

**抱える課題**：

- シャッフル割当を Excel の乱数で手作業で行っており、「自分に当たらない」条件の担保や割当結果の各参加者への個別連絡が煩雑。
- 締切管理・リマインドをすべて手動でこなしており、催促の連絡が負担。
- 提出物と作者の対応表を自分で管理するため、うっかり作者を漏らしてしまうリスクに常に気を遣う。
- 結果発表用のまとめ画像・一覧作成に手間がかかる。

**サイトに求めること**：一連の進行を1画面で管理でき、シャッフルと催促を自動化し、匿名性はシステムが守ってくれること。**主催自身も通常操作では作者を知り得ない**ことで、「自分がうっかり漏らす」不安から解放される。

### 2.2 参加者（プレイヤー）

**ペルソナ**：企画に誘われて／告知を見て参加するイラスト描き。X アカウントを持つ。デザインと作画の両方を担当する。企画を楽しみたいだけで、複雑な手順は避けたい。

**抱える課題**：

- 提出物をどこに・どう出せばよいかがツールごとに異なり分かりにくい。
- 自分がいま何をすべきか（デザイン締切か、作画締切か、投票期間か）が把握しづらい。
- 締切を見落としやすい。
- 「作画する相手のデザイン」以外の情報が見えてしまうと、匿名性・ゲーム性が損なわれる。

**サイトに求めること**：自分のやるべきことと締切が一目で分かり、提出が簡単で、余計な情報（＝作者が分かる情報）が見えないこと。

### 2.3 観覧者（ビューワー）

**ペルソナ**：企画には参加していないが、完成作を見て「誰がデザインしたか」を当てて楽しみたいフォロワー・外部の観客。

**位置づけ**：v1.1 で投票権者に含める。ただし匿名アクセスであるため厳密な本人性は担保できない。観覧者票は**参加者の正解率ランキングとは別枠の「参考集計」**として扱い、二重投票はセッション／ブラウザ単位のゆるい抑止に留める（→ 4.5・8.3）。企画を外部公開するかは主催が企画ごとに設定できるものとする。

## 3. スコープ

### 3.1 MVP に含む範囲

MVP の目的は「1つの誰デザ企画を、サイト内だけで一通り回せること」である。特に誰デザの中核体験である **③シャッフル自動割当・④作画提出・⑤作者当て投票・⑥結果発表** をコアとする。

MVP に含む機能：

- 企画作成と基本的な募集（**参加者ごとの個別トークン付き招待リンク**を最小構成とする）
- 参加者名簿管理（発行トークンが実質的な名簿となる）
- デザイン提出（匿名アップロード）と締切管理
- **③ 自分に当たらない条件つきシャッフル自動割当（提出済みのみ対象）**（コア）
- **④ 作画提出（匿名アップロード）**（コア）
- **⑤ 作者を伏せた一覧と投票（参加者＝厳密1票／観覧者＝参考票）**（コア）
- **⑥ 答え合わせ・正解率ランキングを含む結果発表**（コア）
- サイト内通知（各フェーズ移行・締切前のお知らせ）
- 主催ダッシュボード／参加者ダッシュボード（それぞれの進行管理ビュー、主催側は**匿名ラベル運用**）
- フェーズ状態機械による進行制御（→ 第9章）

### 3.2 MVP に含まない範囲（将来拡張）

以下は重要だが MVP では対象外とし、将来の拡張とする。

- X 連携ログイン（MVP は個別トークン付き招待リンクで代替）
- 承認制・先着制の高度な募集管理（MVP は主催がトークン付きリンクを配る単純方式）
- X DM／メールによる外部通知（MVP はサイト内通知のみ）
- お題（テーマ）ジェネレーター
- まとめ画像の自動生成、X 共有用 OGP 公開ページ
- 作品ギャラリー（過去企画のアーカイブ・横断閲覧）
- 端数・1対多などの例外割当（MVP は**厳密1対1のみ**）
- 複数企画の同時大規模運用、決済・課金など

MVP で「回せること」を最優先し、演出・自動化・外部連携は段階的に追加する方針とする。

## 4. 機能要件

優先度は **Must（MVP 必須）／ Should（できれば MVP、無理なら次点）／ Could（将来拡張）** で示す。ユーザーストーリー（「〜として、〜したい。なぜなら〜」）と合わせて整理する。

### 4.1 企画作成・募集

| ID | 機能 | 優先度 | ユーザーストーリー |
|----|------|--------|------------------|
| F-01 | 企画ページ作成 | Must | 主催として、テーマ・スケジュール（各締切）・説明を設定した企画ページを作りたい。企画の入り口を1つにまとめるため。 |
| F-02 | 個別トークン付き招待リンクによる募集 | Must | 主催として、**参加者ごとに固有のトークン付きリンク**を発行して配りたい。X連携なしで手軽に始めつつ、個人を識別して二重投票・未提出特定・スコアリングを成立させるため。 |
| F-03 | X 連携ログインによる募集 | Could | 参加者として、X アカウントでログインして参加したい。本人確認と通知連携を楽にするため。 |
| F-04 | 参加者名簿管理（承認制／先着） | Should | 主催として、参加希望者を承認制または先着で管理したい。想定人数で企画を締めるため。MVPでは発行済みトークン一覧が実質的な名簿となる。 |

**認証・識別の要点（v1.1 で確定）**：

- MVP は「主催が N 個の**個別トークン**を発行 → 各参加者に個別リンクを配布」を最小構成とする。1本の共有リンクを使い回す方式は、個人を識別できず二重投票防止・未提出特定・スコアリングが破綻するため**採用しない**。
- 個別トークンはアプリ内部で参加者を一意に識別する。この識別情報はシャッフルとスコアリングにのみ内部利用し、作者⇔提出物の対応は主催 UI にも露出させない（→ 第6章）。
- 観覧者は匿名アクセスであり、トークンを持たない。観覧者票の扱いは 4.5 参照。

### 4.2 デザイン提出

| ID | 機能 | 優先度 | ユーザーストーリー |
|----|------|--------|------------------|
| F-05 | 匿名デザインアップロード（画像＋設定文） | Must | 参加者として、キャラのデザイン画像と設定を匿名で提出したい。作者が誰か分からない状態を保つため。 |
| F-06 | デザイン締切の設定・管理 | Must | 主催として、デザイン提出の締切を設定し、締切超過を把握したい。進行を管理するため。 |
| F-07 | 締切前の自動リマインド | Should | 参加者として、締切前に自動でリマインドを受け取りたい。提出忘れを防ぐため。 |

### 4.3 シャッフル自動割当（③・目玉機能）

誰デザの中核であり、最も丁寧に要件化する機能。

| ID | 機能 | 優先度 | ユーザーストーリー |
|----|------|--------|------------------|
| F-08 | 自分に当たらない条件つき自動シャッフル | Must | 主催として、提出済み全員のデザインを「自分のデザインが自分に割り当たらない」条件で自動シャッフルしたい。手作業の乱数抽選をなくすため。 |
| F-09 | 作画対象デザインのみの匿名表示 | Must | 参加者として、自分が作画すべき相手のデザインだけを、作者名を伏せた状態で見たい。他の割当や作者情報が見えると匿名性が壊れるため。 |
| F-10 | 主催用マッチング管理ビュー（匿名ラベル） | Must | 主催として、割当の**成立状況・提出進捗・未提出者**を匿名ラベルで確認し、必要なら再抽選したい。実名や作者は見えないまま破綻がないか管理するため。 |

**シャッフルの詳細要件（v1.1 で厳密化）**：

- **完全撹拌条件**：割当は「デザイン作者 ≠ そのデザインの作画者」を必ず満たすこと（数学的には derangement／完全順列に相当）。
- **成立条件（v1.0 の誤りを修正）**：締切時点で**提出が揃った人数 n ≥ 2 なら derangement は必ず存在する**（n=2 は1通りのみ）。不成立となるのは「提出済みが1人以下」の場合のみ。v1.0 の「参加者2人以下は失敗」という記述は誤りであり、正しくは「提出済み1人以下は不成立」。
- **割当の一意性**：1つのデザインは1人の作画者にのみ、1人の作画者は1つのデザインのみを担当する（**MVP は厳密1対1**。端数・1対多は将来拡張 → 8.2）。
- **未提出者の扱い（確定）**：締切時点で**提出済みの参加者のみ**を割当対象とする。未提出者は自動的に対象外。締切延長・手動調整は主催の任意オプションとして提供しうるが、既定挙動は「提出済みのみで自動割当」。
- **再抽選**：主催が結果を破棄して再シャッフルできること。ただし再抽選しても匿名性が損なわれない設計とする（主催 UI に作者が出ない前提なので、再抽選しても主催は作者を知り得ない）。
- **情報最小化**：参加者側 API・画面には「自分の作画対象デザイン」以外の割当情報を一切返さないこと（匿名性の技術的担保）。
- **アルゴリズム指針**：ランダム置換を生成し derangement 条件を満たすまで再生成する、もしくは条件付き置換を直接構成する。n が小さいため計算量は問題にならない。

### 4.4 作画提出（④）

| ID | 機能 | 優先度 | ユーザーストーリー |
|----|------|--------|------------------|
| F-11 | 匿名作画アップロード | Must | 参加者として、割り当てられたデザインを作画した画像を匿名で提出したい。作画者・作者が投票前に露出しないため。 |
| F-12 | 作画締切の設定・管理 | Must | 主催として、作画提出の締切を設定・管理したい。 |
| F-13 | 締切前の自動リマインド | Should | 参加者として、作画締切前にリマインドを受け取りたい。 |
| F-27 | 作画未提出時のハンドリング | Should | 主催として、作画が締切までに提出されなかった割当（欠番）を把握し、投票一覧から除外するか欠番表示するか選びたい。投票の破綻を防ぐため。 |

**補足**：Assignment は原則 1件の Artwork に対応するが、作画者が未提出だと Artwork が欠ける。この場合は「欠番として投票一覧から除外」を既定とする（→ 4.5・第7章）。

### 4.5 投票：作者当て（⑤）

| ID | 機能 | 優先度 | ユーザーストーリー |
|----|------|--------|------------------|
| F-14 | 作者を伏せた完成作一覧 | Must | 投票者として、作者情報を伏せた完成作の一覧を見て、誰のデザインか推理したい。ゲームとして成立させるため。 |
| F-15 | 各作品のデザイナーを推理する投票（1投票用紙） | Must | 投票者として、各作品について「誰がデザインしたか」を1票ずつ推理して1枚の投票用紙で提出したい。二重投票はできないようにする。 |
| F-16 | 投票締切の管理 | Must | 主催として、投票期間の締切を管理したい。 |
| F-28 | 観覧者投票（参考集計） | Should | 観覧者として、公開された企画で作者当てに参加したい。ただし参加者ランキングとは別枠で集計されること。 |

**投票の詳細要件（v1.1 で明確化）**：

- **用語の整理**：「1人1票」は誤解を招くため廃し、正しくは「**1投票者＝投票用紙1枚**、その用紙内で各作品のデザイナーを1名ずつ推理する」。「二重投票禁止＝1投票者につき投票用紙は1枚」。
- **投票形式**：各作品ごとに、参加者候補リストからデザイナーを1名選ぶ形式（MVP）。全体マッチングを組ませる形式は将来検討。
- **投票権者（確定）**：参加者＋観覧者。参加者はトークンで厳密に1枚に制限。観覧者は匿名のためセッション／ブラウザ単位のゆるい重複抑止に留め、票は**参考集計**として参加者票と分離する。
- **自己票の扱い（確定）**：参加者は自分のデザインの正解を必ず知っているため、**自分のデザインに対する自身の推理はスコア集計対象外**とする（実質的な自己投票禁止）。UI 上は自作品を選択不可、または集計時に除外。
- **やり直し（確定）**：**投票締切前は再提出（変更）可**。締切到達時点の内容で確定。
- **正誤判定の経路**：Vote は対象 Artwork に紐づくが、正解はその作品の元 Design の作者 Participation。判定は `Vote →（Artwork→Assignment→Design→作者Participation）` の経路で行う（→ 第7章）。

### 4.6 結果発表（⑥）

| ID | 機能 | 優先度 | ユーザーストーリー |
|----|------|--------|------------------|
| F-17 | 答え合わせ表示 | Must | 参加者として、各作品の本当のデザイナー（と作画者）を答え合わせで見たい。 |
| F-18 | 正解率ランキング（参加者対象） | Must | 参加者として、誰が何問正解したかのランキングを見たい。企画の締めを盛り上げるため。観覧者票は別枠の参考集計として併記する。 |
| F-19 | まとめ画像／シートの自動生成 | Could | 主催として、結果まとめ画像やシートを自動生成したい。発表・共有の手間を省くため。 |
| F-20 | X 共有用 OGP 公開ページ | Could | 主催として、結果を X で共有できる OGP 付き公開ページを出したい。企画を外部に見せるため。 |

**スコアリング仕様（v1.1 で明確化）**：

- 1作品のデザイナーを正しく当てるごとに +1 点（MVP は単純加点）。
- 参加者ランキングは**自作品を除外**した母数で正解率を算出（例：n 作品中、自作 1 を除く n-1 作品が対象）。
- 欠番（作画未提出）作品は投票・採点対象から除外。
- 同点時の順位付け・表示ルールは設計フェーズで確定（暫定：同順位表示）。
- 観覧者票は参加者ランキングに混ぜず、参考値として集計・表示する。

## 6. 非機能要件

### 6.1 匿名性・セキュリティ（最重要）

- **作者情報の分離**：提出物と作者の紐付けは投票・結果発表フェーズまでシステム内部でのみ保持し、参加者・観覧者・**主催**のいずれの通常画面・API にも露出しないこと。
- **主催に対する匿名性（v1.1 で確定）**：主催の通常操作（ダッシュボード・マッチング管理）では作者⇔提出物の対応を**全期間秘匿**する。管理は「デザインA／参加者#3」等の**安定した匿名ラベル**で行う。作者の開示は**緊急時のみ・監査ログ付き**の例外操作に限定する（F-29）。
- **作画者の匿名性**：デザイナーだけでなく作画者も結果発表まで秘匿する（→ 1.4 設計原則）。
- **割当情報の最小開示**：シャッフル割当は「自分の作画対象」以外を参加者に返さない（4.3 参照）。主催にも実名割当を返さない。
- **メタデータ経由の漏洩防止**：アップロード画像の Exif など、作者を推測できるメタデータを除去すること。ファイル名・提出順・タイムスタンプから作者が推測されないよう配慮する。完成作一覧は**固定のシャッフル順**で表示し、提出順が漏れないようにする。
- **絵柄による身バレは受容リスク**：絵柄・画風からの推測は原理的に防げないため、システムでの防御対象外とし「受容リスク」として扱う（これはゲーム性の一部でもある）。
- **アクセス制御**：企画は招待された参加者（トークン保持者）と、公開設定時の観覧者のみアクセスでき、**推測困難な ID（UUID/ランダムslug）**で URL 推測による他企画・他人提出物への到達を防ぐこと。
- **投票の公正性**：参加者は1投票用紙に厳密制限。観覧者はセッション／ブラウザ単位のゆるい抑止に留め、参考集計として分離する。

### 6.2 想定利用規模・性能

- 1企画あたり数人〜十数人の小規模。同時アクセスも小さい。高トラフィック前提の大規模スケーリングは MVP では不要。
- ただし複数企画が並行して存在しうる点は考慮する。
- 公開設定時は観覧者アクセスが一時的に増える可能性を考慮する（結果発表・投票ページの軽量化）。
- 画面表示・提出処理はストレスのない応答速度（体感数秒以内）を目安とする。

### 6.3 画像アップロード

- 対応形式：一般的な画像形式（例：PNG／JPEG。必要に応じて上限枚数・解像度を定義）。
- 容量上限：イラスト用途を踏まえ1枚あたりの上限を設定する（具体値は要検討 → 8.6）。
- アップロード時に **Exif 等メタデータを除去**する処理を必須とする。
- サムネイル生成・表示最適化を検討する。

### 6.4 対応環境・アクセシビリティ

- **モバイル対応必須**：参加者の多くはスマホから提出・投票する想定。レスポンシブ対応とする。
- モダンブラウザ対応。過度に古い環境は対象外で可。
- 文言・UI は日本語を基本とする。

### 6.5 データ保全・運用

- 提出物・投票結果のバックアップ方針を定める。
- 企画終了後のデータ保持期間・削除ポリシーを定める（アーカイブ機能との関係 → 8.8）。
- 緊急開示操作は監査ログとして永続記録する。

## 7. データモデル概略

主要エンティティと関係の概略（詳細な属性は設計フェーズで確定）。

### 7.1 主要エンティティ

- **User（ユーザー）**：主催を表す。参加者はトークンベースの一時識別でも可（→ 8.1）。認証情報、表示名など。
- **Project（企画）**：1つの誰デザ企画。テーマ、各フェーズの締切、募集方式、公開範囲、**現在のフェーズ状態**（→ 第9章）、主催を持つ。
- **InviteToken（招待トークン）**：主催が発行する参加者ごとの一意トークン。Participation と1対1で対応し、参加者の識別に用いる。
- **Participation（参加）**：ある企画へのある参加者の参加状態（承認待ち／参加中など）。InviteToken に紐づく。
- **Design（デザイン提出）**：参加者が提出したデザイン（画像＋設定文）。作者（Participation）への紐付けは秘匿対象。
- **Assignment（割当）**：シャッフルで決まる「あるデザインを、ある作画者が担当する」対応。デザインと作画者（Participation）を結ぶ。「作者 ≠ 作画者」制約を保証する。
- **Artwork（作画提出）**：割当に基づいて提出された完成作画。1つの Assignment に対応する（作画未提出時は欠番）。
- **Vote（投票）**：ある投票者が、ある Artwork について「デザイナーは誰か」を推測した1票。投票用紙（Ballot）単位でまとめる。
- **Ballot（投票用紙）**：1投票者の一連の Vote をまとめる単位。参加者は Participation に、観覧者は匿名セッションに紐づく。参加者は1枚に制限。
- **Notification（通知）**：ユーザー宛のサイト内通知。
- **AuditLog（監査ログ）**：緊急開示など、匿名性に関わる例外操作の記録。

### 7.2 関係（概略）

- User 1 —— * Project（主催として）
- Project 1 —— * InviteToken 1 —— 1 Participation
- Participation 1 —— * Design（基本は1つ）
- Design 1 —— 1 Assignment 1 —— 1 Participation（作画者）　※「作者 Participation ≠ 作画者 Participation」制約
- Assignment 1 —— 0..1 Artwork（作画未提出時は欠番）
- Ballot 1 —— * Vote * —— 1 Artwork
- Ballot * —— 0..1 Participation（参加者票）／匿名セッション（観覧者票）
- User/Participation 1 —— * Notification
- Project 1 —— * AuditLog

### 7.3 匿名性に関わるモデル上の注意

- Design／Artwork と「作者 Participation」の紐付けは、投票・結果発表フェーズまでアプリケーション層でアクセス制御し、参加者・観覧者・**主催**向けのレスポンスに含めないこと。
- 「作者当ての正解データ」＝ Design と作者 Participation の対応であり、これが投票フェーズ中に漏れない設計を最優先とする。
- 正誤判定は `Vote →（Artwork→Assignment→Design→作者Participation）` の経路でサーバ側のみ実行する。
- 作画者（Assignment の作画者 Participation）も結果発表まで秘匿する。

## 8. 未確定事項・要検討リスト（v1.1 で一部確定）

### 8.1 認証・アカウント

- 【方針確定】MVP は**参加者ごとの個別トークン付き招待リンク**を採用。1本の共有リンク使い回しは不採用。
- 【要検討】参加者にアカウント登録まで求めるか、トークン単位の一時識別で済ませるか。MVP は一時識別を基本とし、X 連携は将来拡張。
- 【要検討】トークン紛失・再発行のフロー。

### 8.2 シャッフル割当の仕様

- 【方針確定】MVP は**厳密1対1（derangement）のみ**。端数・1対多は将来拡張。
- 【方針確定】未提出者は**締切時点の提出済みのみで自動割当**（未提出は自動除外）。締切延長・手動調整は任意オプション。
- 【要検討】再抽選の回数上限（現状は主催 UI に作者が出ないため匿名性リスクは低いが、上限を設けるか）。

### 8.3 投票の仕様

- 【方針確定】投票権者は**参加者＋観覧者**。観覧者票は参考集計として分離。
- 【方針確定】投票形式は**作品ごとに候補から1名選択**（MVP）。
- 【方針確定】**締切前は変更可**。
- 【方針確定】参加者の**自作品は自己投票禁止（集計除外）**。
- 【要検討】観覧者の重複投票抑止をどの程度厳密にするか（セッション／Cookie／IP のいずれ・組合せ）。

### 8.4 主催の権限と匿名性の両立

- 【方針確定】**全期間秘匿＋緊急時のみ監査ログ付き開示**。通常運用では主催も作者を見られない。
- 【要検討】緊急開示の発動条件・粒度（全件開示か個別開示か）、ログの保持期間と閲覧権限。

### 8.5 ホスティング・インフラ

- 【要検討】ホスティング形態（マネージドPaaS／サーバレス／自前サーバ）の選択肢と、想定規模・運用体制・コストのバランス。小規模前提のコスト最適化をどこまで反映するか。

### 8.6 画像ストレージ

- 【要検討】画像の保存先（オブジェクトストレージ／CDN 付きサービス／DB 内保存）。
- 【要検討】1枚あたり容量上限・対応形式・保存期間・削除ポリシー。
- 【方針確定】Exif 等メタデータ除去はアップロード時に必須実行（具体手段は設計時に選定）。

### 8.7 通知チャネル

- 【方針確定】MVP は**サイト内通知のみ**。
- 【要検討】外部通知（X DM／メール／Web Push）の実装範囲。X DM は API 制約・到達性の問題があるため、メール併用や Web Push を優先検討。

### 8.8 データ保持・アーカイブ

- 【要検討】企画終了後のデータ保持期間、参加者による削除依頼への対応、作品ギャラリー（アーカイブ）公開の可否と公開範囲。

### 8.9 技術スタック（参考論点）

- フロントエンド／バックエンド／DB の選定は本要件確定後に行う。選定にあたっては「モバイル対応必須」「画像アップロード」「**匿名性を守るサーバ側アクセス制御のしやすさ**」「小規模運用でのコスト」を評価軸とする。現段階では特定技術を確定しない。

## 9. フェーズ状態機械（v1.1 で新設）

企画は以下の状態を順に遷移する。各遷移のトリガーと、その状態で許可される操作・匿名性の境界を定義する。これが実装の背骨となる。

| フェーズ | 説明 | 遷移トリガー | 主な許可操作 | 匿名性の境界 |
|---------|------|-------------|-------------|-------------|
| `Draft`（下書き） | 主催が企画を設定中 | 主催が公開 | 企画設定編集、トークン発行 | — |
| `Recruiting`（募集） | 参加者を集める | 主催が募集締切 or デザイン期間開始 | トークン配布、参加受付 | — |
| `DesignSubmission`（デザイン提出） | 参加者がデザインを提出 | デザイン締切到達 → 主催がシャッフル実行 | デザイン匿名提出、リマインド | デザイン作者は秘匿開始 |
| `Shuffling`（シャッフル） | 提出済みで割当を生成 | 主催が割当確定 | derangement 生成、再抽選、匿名ラベル管理 | 主催にも実名割当を出さない |
| `ArtworkSubmission`（作画提出） | 各自が割当デザインを作画 | 作画締切到達 | 作画対象の匿名表示、作画匿名提出 | 割当・作画者ともに秘匿 |
| `Voting`（投票） | 作者当て投票 | 投票締切到達 | 匿名一覧表示、投票（締切前変更可） | 作者・作画者・割当すべて秘匿 |
| `Result`（結果発表） | 答え合わせ・ランキング | （終端） | 答え合わせ、ランキング表示 | 秘匿解除（作者・作画者を開示） |

**遷移の原則**：

- フェーズは原則として前方向にのみ進む。巻き戻しは主催の例外操作とし、匿名性を損なわない範囲に限る。
- 各フェーズ移行時に該当参加者へサイト内通知を送る（F-23）。
- **秘匿解除は `Result` 到達時のみ**。それ以前のいかなるフェーズでも、作者・作画者・割当の対応をクライアントへ返してはならない（緊急開示 F-29 を除く）。
- `Shuffling` は「デザイン締切 → 主催が明示的に実行」という手動ゲートとし、提出が揃った参加者のみを対象とする。
